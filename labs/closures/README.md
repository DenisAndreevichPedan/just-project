## Выводы по кейсам

**Case 1** — каждый вызов `createMessagePrinter` создаёт своё замыкание со своим `message`. `printForAlex` и `printForSam` не пересекаются.

**Case 2** — `likesCounter` и `viewsCounter` — два независимых замыкания: у каждого свой `count` и свой `label`. Вызов одного счётчика не сбрасывает другой.

**Case 3** — `var` имеет function scope, поэтому все три функции смотрят на **одну** переменную `i`. К моменту вызова колбэков цикл уже завершился и `i === 3`.

**Case 4** — `createdTickets` меняется, а `message` был собран один раз при создании (`"Created tickets: 0"`). `printStats` читает старую строку (stale closure), `printFreshStats` — живую переменную.

## Самопроверка (ответы)

1. **Closure** — функция помнит переменные из внешней области видимости даже после того, как внешняя функция уже завершилась.
2. **Counter помнит значение** — `count` живёт в замыкании возвращённой функции; каждый вызов `increment` читает и обновляет ту же переменную.
3. **`var` в цикле** — все колбэки ссылаются на одну переменную, а не на значение конкретной итерации.
4. **`let` в цикле** — на каждой итерации создаётся новая привязка, поэтому каждый колбэк замыкается на своё значение.
5. **Stale closure** — функция использует старое значение, хотя переменная уже изменилась (часто из-за «замороженного» снимка вроде заранее собранной строки).
6. **В React** — в `useEffect` / колбэках с пустым массивом зависимостей `[]`, в `setInterval`, обработчиках событий, которые «застряли» на props/state старого render.

## Задача на counter (без GPT)

```js
function createCounter() {
  let count = 0;

  return {
    increment() {
      count += 1;
    },
    getValue() {
      return count;
    },
    reset() {
      count = 0;
    },
  };
}

const firstCounter = createCounter();
const secondCounter = createCounter();

firstCounter.increment();
secondCounter.increment();
secondCounter.increment();

secondCounter.getValue(); // 2
firstCounter.getValue(); // 1
```

Два вызова `createCounter()` не влияют друг на друга — у каждого своё замыкание.

## Итоги

**Главное правило №1:**  
Closure — это доступ внутренней функции к переменным внешней функции после её завершения.

**Главное правило №2:**  
Замыкание может хранить **живую** переменную (`count`) или **устаревший снимок** (`message`) — от этого зависит, увидишь ли ты актуальное значение.

**Где ошибся и почему:**  
Не ошибся — все 4 прогноза совпали с фактом после запуска `node labs/closures/week5-closures.js`.
