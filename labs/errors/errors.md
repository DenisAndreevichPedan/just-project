# Неделя 4 — заметка: распространение ошибок (error propagation)

Краткий конспект для критерия «можешь объяснить устно и письменно».

## 1. Ошибка в синхронном коде

Здесь всё просто: код внутри `try` выполняется **подряд, без пауз**. Если в этот момент что-то «ломается», срабатывает ближайший `catch`.

**Где `throw` видно в коде** — ты пишешь его сам:

```js
try {
  throw new Error("что-то пошло не так");
} catch (e) {
  // сюда попадём: catch поймал эту ошибку
}
```

**Где `throw` не написан у тебя в файле** — его делает функция при ошибке (внутри себя она как раз делает `throw`). Например, битый JSON:

```js
try {
  JSON.parse("{"); // внутри JSON.parse при ошибке парсинга выбрасывается исключение
} catch (e) {
  // поймаем — пока мы ещё «внутри» try, выполнение не успело выйти наружу
}
```

Итого: `try/catch` срабатывает, когда ошибка возникает **в том же обычном прогоне кода**, пока выполняется блок `try`. Слово `throw` означает «создать ошибку и немедленно прервать текущий участок кода»; дальше управление передаётся в `catch`, если он есть.

## 2. Ошибка внутри Promise

`throw` в **исполнителе** `new Promise((resolve, reject) => { ... })` или `reject(...)` делает промис отклонённым. Окружающий синхронный `try/catch` вокруг `new Promise(...)` **обычно не перехватывает** такой выброс: движок переводит его в состояние reject у созданного промиса (в лабе в Node это видно как `unhandledRejection`, если на промис не повесили `.catch`).

`throw` в колбэке `.then` после микротаска — тоже становится **отклонением цепочки**; ловить нужно `.catch` / `await` в `try/catch` на этой цепочке.

## 3. Ошибка внутри `async`

Функция с `async` **всегда** возвращает Promise.

- `throw` внутри `async` эквивалентен `return Promise.reject(...)`.
- Внешний `try/catch` **не** поймает «сырой» вызов `asyncFn()` без `await`: к моменту отклонения промиса синхронный блок `try` уже завершён.

Чтобы поймать: `await asyncFn()` внутри `try/catch` или `.catch()` на возвращённом промисе.

## 4. Отклонённый Promise без `.catch` / без `await`

Если промис отклонился и на нём **нет** обработчика отказа, среда фиксирует **unhandled rejection** (в браузере — событие `unhandledrejection`, в Node — `unhandledRejection`). Это не «тихий» сбой: его нужно устранять, добавляя обработку в том месте, где цепочка обрывается.

## 5. Почему `try/catch` не ловит ошибку из `setTimeout` «напрямую»

Колбэк `setTimeout` ставится в **очередь макрозадач** и выполняется **позже**, в **новом** турне event loop, с **новым** стеком вызовов.

`try/catch` охватывает только синхронный код до выхода из `try`. К моменту выполнения колбэка таймера выполнение уже вышло из блока `try`, поэтому исключение из колбэка **не связывается** с тем `catch`.

Как ловить: `try/catch` **внутри** колбэка таймера, или оборачивать асинхронную работу в Promise и использовать `.catch` / `await` + `try/catch`.
